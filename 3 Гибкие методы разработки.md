## **1. Введение в гибкие методологии**

### **1.1. Понятие Agile**
**Agile** — это итеративный подход к управлению проектами и разработке ПО, который помогает командам быстрее предоставлять ценность для клиентов без лишних затрат.

### **1.2. Манифест гибкой разработки (Agile Manifesto)**
Четыре ключевые ценности:
1. **Люди и взаимодействие** важнее процессов и инструментов
2. **Работающий продукт** важнее исчерпывающей документации
3. **Сотрудничество с заказчиком** важнее согласования условий контракта
4. **Готовность к изменениям** важнее следования первоначальному плану

### **1.3. 12 принципов Agile**
- Наивысшим приоритетом является удовлетворение клиента
- Изменения требований приветствуются даже на поздних стадиях
- Частая поставка работающего программного обеспечения
- Ежедневное сотрудничество заказчика и разработчиков
- Проекты строятся вокруг мотивированных личностей

## **2. Основные гибкие методологии**

### **2.1. Scrum**

#### **Роли в Scrum:**
- **Product Owner** — представляет интересы заказчика
- **Scrum Master** — обеспечивает соблюдение процессов
- **Development Team** — команда разработчиков (5-9 человек)

#### **Артефакты Scrum:**
- **Product Backlog** — приоритизированный список требований
- **Sprint Backlog** — задачи на текущий спринт
- **Increment** — работающий продукт после спринта

#### **События Scrum:**
- **Sprint Planning** — планирование спринта (2-4 недели)
- **Daily Stand-up** — ежедневные 15-минутные встречи
- **Sprint Review** — демонстрация результатов
- **Sprint Retrospective** — анализ процесса

### **2.2. Kanban**

#### **Принципы Kanban:**
- Визуализация рабочего процесса
- Ограничение работы в процессе (WIP)
- Управление потоком
- Прозрачность процессов

#### **Kanban-доска:**
```
| Бэклог | В работе | Ревью | Готово |
|--------|----------|--------|---------|
| Задача1| Задача2  | Задача3| Задача4 |
```

### **2.3. Extreme Programming (XP)**

#### **Практики XP:**
- **Парное программирование** — два разработчика за одним компьютером
- **Test-Driven Development (TDD)** — разработка через тестирование
- **Непрерывная интеграция** — частые сборки проекта
- **Рефакторинг** — улучшение структуры кода
- **Частые выпуски** — короткие циклы выпуска

## **3. Гибкие практики разработки**

### **3.1. User Stories**
```
Как [роль], я хочу [функциональность], чтобы [получить пользу]
```
**Пример:** "Как пользователь, я хочу фильтровать товары по цене, чтобы быстро найти подходящий вариант"

### **3.2. Story Points**
- Относительная оценка сложности задач
- Использование последовательности Фибоначчи: 1, 2, 3, 5, 8, 13
- Planning Poker для коллективной оценки

### **3.3. Definition of Done (DoD)**
Критерии завершенности задачи:
- Код написан и протестирован
- Code review пройден
- Документация обновлена
- Тесты автоматизированы

## **4. Гибкие框架и масштабирования**

### **4.1. SAFe (Scaled Agile Framework)**
- Для крупных предприятий
- Выравнивание стратегии и исполнения
- Координация множества команд

### **4.2. LeSS (Large Scale Scrum)**
- Масштабирование Scrum для нескольких команд
- Сохранение простоты и гибкости

### **4.3. Nexus**
- Framework для координации 3-9 Scrum-команд
- Фокус на минимизацию зависимостей

## **6. Метрики в гибкой разработке**

### **6.1. Velocity**
- Количество story points, завершенных за спринт
- Используется для прогнозирования

### **6.2. Burn-down Chart**
- Визуализация оставшейся работы в спринте
- Помогает отслеживать прогресс

### **6.3. Cumulative Flow Diagram**
- Показывает распределение задач по статусам
- Выявляет узкие места в процессе

## **7. Преимущества и вызовы Agile**

### **7.1. Преимущества**
- Быстрая адаптация к изменениям
- Раннее получение обратной связи
- Повышение прозрачности процесса
- Улучшение качества продукта
- Высокая мотивация команды

### **7.2. Вызовы и ограничения**
- Требует высокой дисциплины команды
- Сложности с долгосрочным планированием
- Не подходит для жестко регламентированных проектов
- Зависимость от вовлеченности заказчика

---

## **8. Практическое применение**

### **8.1. Внедрение Agile в организации**
**Этапы внедрения:**
1. Обучение и осознание
2. Пилотный проект
3. Масштабирование
4. Непрерывное улучшение

### **8.2. Роли и ответственности**
**Product Owner должен:**
- Формировать и приоритизировать бэклог
- Быть доступным для команды
- Принимать решения по продукту

**Scrum Master должен:**
- Обучать команду процессам
- Устранять препятствия
- Фасилитировать встречи

## **9. Практическое задание**

### **Задача:** Разработать план первого спринта для проекта "Система управления задачами"

**Требуется:**
1. Сформировать Product Backlog из 10 User Stories
2. Провести оценку сложности методом Planning Poker
3. Выбрать задачи для первого спринта (емкость: 20 story points)
4. Определить Definition of Done
5. Создать план ежедневных стендапов

**Пример User Stories:**
```
1. Как пользователь, я хочу создавать новые задачи, чтобы вести список дел
2. Как пользователь, я хочу назначать приоритет задачам, чтобы фокусироваться на важном
3. Как пользователь, я хочу фильтровать задачи по статусу, чтобы видеть актуальные
```
